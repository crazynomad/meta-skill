## 二、冠军评测

### ship(工作流型)— 🟢 健康标杆

轴 2-4:判断密集混合 / 爆炸半径**高**(push、建 PR、版本变更)/ 显式+自动触发

| 层 | 结果 | 证据 |
|----|------|------|
| L0 | ✓ | 73.5KB < 100KB。**v1.57.7 baseline 时为 174KB,Token-reduction Phase B 砍半** — L0 预算追踪→重构波次回路真实运转过的直接证据 |
| L1 | ○ | triggers 含 "push to main",有误触风险(用户可能只想推送),应入负例集 |
| L2/L3 | ✓✓ | 全仓最强:gate ×4(base-branch / local-workflow / coverage-audit / triage)+ periodic ×3(含 ~$3/run 真实 PTY 幂等测试)+ golden file 夹具 |
| L4 | ✓ | preamble-tier 4,遥测 + operational learning 捕获 + GBRAIN_CONTEXT_LOAD |

### codex(工具包装型)— 🟡 体积与描述失衡

轴 2-4:判断中等 / 爆炸半径低(读为主二次意见)/ 自动触发("second opinion")

| 层 | 结果 | 证据 |
|----|------|------|
| L0 | ⚠ | 85.2KB,已用掉上限 **85%** — 工具包装却比 ship 还重 |
| L1 | ⚠ | description 全文 *"OpenAI Codex CLI wrapper — three modes. (gstack)"* — 三个模式是什么没说,触发全靠 triggers 列表硬扛 |
| L2/L3 | ✓ | gate ×4 全叫 `codex-offered-*` — 测的正是「该不该递出二次意见」的**触发边界**,说明此处踩过坑,L1 问题被 L2 测试钉住;另有 periodic 级 codex-e2e(外部服务依赖,分级正确) |
| L4 | ○ | 有 PREAMBLE(tier 3),无 LEARNINGS 宏 |

建议:description 补一句三模式说明(Review / Challenge / Consult);列入瘦身候选。

### guard(行为塑造型)— 🟠 架构对,观测盲

轴 2-4:确定性强(规则拦截)/ 爆炸半径为**负**(职责是降低别人的爆炸半径)/ 显式触发

| 层 | 结果 | 证据 |
|----|------|------|
| L0 | ✓ | 3.3KB 极轻 |
| 架构 | ✓✓ | 不靠模型自觉:PreToolUse hook 跑 `check-careful.sh` 确定性拦截,脚本有单元测试(`careful/test/hook-scripts.test.ts`)— 把「模型会不会听话」转化为「代码逻辑对不对」 |
| L2/L3 | ✗ | **touchfiles 零条目** — diff-based 选择永远不会为它触发任何 eval;「hook 真的装上了、真的拦住了」无 E2E 验证。安全护栏应配 gate 级接线测试(agent 尝试 rm -rf → 断言被警告) |
| L4 | ✗ | 无 PREAMBLE,零遥测。**安全 skill 自己最没有可观测性**:拦截次数、用户 override 率全无记录,而行为塑造型的 L4 在框架矩阵中是 ✦✦ |

### retro(元技能型)— 🟢 健康,注意体积

轴 2-4:判断密集 / 读为主 + 写 learnings / 显式触发

| 层 | 结果 | 证据 |
|----|------|------|
| L0 | ⚠ | 87.4KB,全仓第二接近上限(87%) |
| L1 | ○ | description 极简("Weekly engineering retrospective"),靠 triggers 补偿 |
| L2/L3 | ✓ | periodic 主测试 + gate 级 base-branch;含事故驱动回归(#1624 stale-base 守卫) |
| L4 | ✓✓ | 全仓接线最全:PREAMBLE + LEARNINGS_SEARCH/LOG + GBRAIN load/save — 合理,它是改进回路的人工裁决端 |

## 四、回灌框架的修订(已并入 v1.1)

1. **L0↔L1 张力规则**:description 瘦身省 catalog token 但伤触发查全,
   瘦身后必须重跑 L1(codex/retro 为证)
2. **行为塑造型优先 hook 化**,但 hook 化后仍需 E2E 验证接线(guard 为证)
3. **L0 新增检查:测试注册表完备性** — 每个 skill 必须在 touchfiles 有条目
   或显式豁免(guard/careful 盲区为证)
4. **分类空格有信息量** — 审计报告保留空格,不强行填类
